# Large Update Design Guide

## 概要

業務システムでは、単一画面で完結する更新処理だけでなく、**非常に大規模な更新処理**が存在する。

例えば次のようなケースである。

- 入力項目が非常に多い  
- 入力画面が複数ページに分割されている  
- 入力完了までに複数回の画面遷移が発生する  

このような更新処理は、物理的には複数画面で構成されるが、業務的には **1つの更新処理**として扱われる。

本ドキュメントでは、SPA において **複数画面に分割された更新処理をどのように設計するか**を説明する。

---

# 通常更新との違い

Router‑First SPA Architecture では、通常の更新画面は **Snapshot Form Screen** として実装される。

```
URL
↓
Loader
↓
Snapshot
↓
Form Initial State
↓
Draft State
```

実装イメージ

```
useLoaderData()
↓
useForm({ defaultValues: snapshot })
```

このモデルでは

- Loader が取得した Snapshot がフォーム初期値になる  
- 編集対象は URL によって一意に決まる  
- Navigation は Data Fetch Transaction として扱われる  

この方式は **単一画面で完結する更新処理**に適している。

---

# 複数画面更新で発生する課題

巨大更新では **1つの更新処理が複数画面に分割される**。

```
1 Update Operation
↓
N Screens
```

この構造では次の課題が発生する。

---

## 編集状態が画面遷移で失われる

React の Component State は **URL を跨ぐ Navigation では保持されない**。

```
Screen A
↓ navigation
Screen B
↓ navigation
Screen C
```

このとき

```
Component State
```

は画面遷移のたびに破棄される。

そのため

- UI 状態  
- フォーム入力状態  

をどのように保持するかを設計する必要がある。

---

## UI State の維持

UI の表示状態は **URL に保存する**。

```
UI State = URL
```

例

```
/order/edit?_step=2&_tab=items
```

この設計により

- 戻る操作  
- 画面再表示  
- 画面遷移  

において UI 状態を安全に復元できる。

この原則は巨大更新に限らず **すべての画面で適用される基本ルール**である。

---

## フォーム入力状態の維持

フォーム入力状態は **URL を跨いで維持することができない**。

そのため複数画面更新では

```
フォーム入力状態をどこに保存するか
```

を設計する必要がある。

---

# 作業テーブル方式

巨大更新では **対象データを直接編集せず、作業テーブルを経由して更新する方式**を採用する。

```
Production Table
↓
Draft Table（作業テーブル）
↓
Final Commit
```

手順

1 編集開始時に対象データを作業テーブルにコピーする  
2 更新画面は作業テーブルに対して読み書きを行う  
3 各画面を離れる際に作業テーブルを更新する  
4 最終確定時に作業テーブルの内容を本番テーブルへ反映する  

作業テーブルは **ユーザー専用の編集領域**として扱う。

---

# 更新フロー

複数画面更新では **画面単位で作業テーブルを更新する**。

```
Screen Input
↓
Validate
↓
Update Draft Table
↓
Next Screen
```

この処理が画面ごとに繰り返される。

```
Screen Input
  ↓
  Validate
  ↓
  Update Draft Table
  ↓
  Next Screen
↓
Screen Input
  ↓
  Validate
  ↓
  Update Draft Table
  ↓
  Next Screen
↓
Final Commit
```

---

# 作業テーブル方式の利点

## 編集状態を安全に保持できる

フォーム入力は作業テーブルに保存されるため  
画面遷移によって失われない。

---

## Snapshot Form モデルと整合する

各画面では

```
Loader
↓
Draft Snapshot
↓
Form Initial State
```

という構造になる。

つまり

```
useLoaderData()
```

をフォーム初期値として利用できる。

---

## フォーム再初期化が問題にならない

画面遷移後も

```
Loader Data = Draft Table
```

であるため

```
Form Reset
```

は業務上正しい動作になる。

---

## 業務チェックを段階的に実行できる

各画面を離れる際に

```
業務ルールチェック
```

を実行できる。

そのため

- エラーを早期に検出できる  
- 修正もその画面で行える  

---

# 更新セッション

作業テーブルを用いた一連の編集処理を **更新セッション** と呼ぶ。

```
Edit Start
↓
Create Draft Table
↓
Edit Screens
↓
Update Draft Table
↓
Final Commit
↓
Edit End
```

つまり

```
1 更新処理
↓
1 更新セッション
↓
複数画面
```

という構造になる。

---

# 更新セッションの実現方法

更新セッションを実装するために **更新セッションID** を利用する。

更新セッションIDは、現在進行中の更新処理を識別するための識別子である。

---

## 更新セッションIDの発行

更新処理開始時にサーバーは更新セッションIDを発行する。

同時にサーバー側のデータベースに **更新セッションの開始を記録する。**

```
Edit Start
↓
Generate session_id
↓
Insert UpdateSession(session_id, user_id)
↓
Create Draft Table
```

---

## 更新セッションIDの共有

更新セッションIDは **URL の search parameter** として画面間で共有する。

例

```
/order/edit?session=abc123
```

これにより

- React Component State に依存せず  
- 画面遷移を跨いでも  
- 同一の更新セッションを識別できる  

---

## 更新処理時の検証

更新処理では必ず更新セッションIDをサーバーへ送信する。

サーバーは

```
session_id が有効か
```

を確認する。

```
if session_id not exists
  error
```

これにより

- 完了済み更新
- 不正URL
- 履歴アクセス

を検出できる。

---

## 更新セッションの終了

更新セッションは以下のタイミングで終了する。

### 最終更新

```
Final Commit
↓
Draft → Production
↓
Delete UpdateSession
```

この時点で更新セッションIDは無効になる。

そのため履歴やURLによる再アクセスは

```
invalid session
```

としてエラーになる。

---

### ログアウト

ログアウト時には未完了の更新セッションを削除する。

```
Logout
↓
Delete UpdateSession(user_id)
```

---

## 履歴復帰による再編集

更新途中で画面を離脱した場合でも、更新セッションIDが有効であれば編集を再開できる。

これはブラウザ履歴やURL操作を完全に制御できないため、**設計上許容する挙動**とする。

必要に応じて

- セッション有効期限  
- 最終アクセス時間  

などを用いてセッションを無効化することも可能である。

ただし巨大更新では入力時間が長くなる可能性があるため、有効期限の設定は慎重に検討する必要がある。

---

# 最終更新

すべての入力が完了した時点で

```
Draft Table
↓
Production Table
```

へ反映する。

このとき

```
version
optimistic lock
```

を用いて整合性を確認する。

---

## 最終更新時の整合性確認

更新セッションIDによって更新処理の有効性は確認できるが、**画面遷移順序までは保証できない。**

そのため最終更新時には

```
必要なデータが揃っているか
```

を確認する。

各画面では既に業務チェックが実行されているため

```
必須レコードの存在確認
```

のみで全体整合性を保証できる。

---

# まとめ

複数画面に分割された更新処理では、**作業テーブル方式**を採用する。

基本ルール

1 編集開始時に対象データを作業テーブルにコピーする  
2 更新画面は作業テーブルを読み書きする  
3 各画面を離れる際に作業テーブルを更新する  
4 最終画面で作業テーブルを本番テーブルへ反映する  

この設計により

- フォーム入力状態の安全な保持  
- Snapshot Form モデルとの整合  
- 段階的な業務チェック  

を実現できる。
