# BFF API Response Conventions

## 1. 目的

本ドキュメントでは、BFF (Backend for Frontend) が返す HTTP レスポンスの推奨仕様を定義する。

目的は次の通り。

- クライアントとサーバー間の意味論を統一する  
- HTTP ステータスコードを正しく利用する  
- クライアント側のエラーハンドリングを単純化する  

本仕様は推奨設計であり、すべての API に適用することを意図している。

---

# 2. 基本方針

BFF は次の方針でレスポンスを返す。

1. HTTP ステータスコードは意味論に従って使用する  
2. ビジネスエラーは HTTP ステータスコードで表現する  
3. クライアントはステータスコードに依存しすぎない  

---

# 3. 推奨ステータスコード

更新処理に関係する代表的なステータスコードを以下に示す。

| 状態 | ステータス | 意味 |
|-----|-------------|------|
| 正常 | 200 / 204 | 処理成功 |
| 楽観的排他ロック | 409 Conflict | データ競合 |
| 削除済みデータ | 410 Gone | 対象データは削除済み |
| 認証エラー | 401 Unauthorized | セッション切れ |
| 権限エラー | 403 Forbidden | 操作権限なし |
| 入力エラー | 422 Unprocessable Entity | バリデーションエラー |
| サーバーエラー | 500 Internal Server Error | 予期しないエラー |

---

# 4. 楽観的排他ロック

更新処理で version 不一致が発生した場合、BFF は次を返す。

```
409 Conflict
```

これは

```
リソース状態の競合
```

を意味する。

クライアントは

```
更新競合メッセージ
```

をユーザーに表示する。

---

# 5. 削除済みデータ

更新画面などで指定された主キーのデータが削除されている場合、BFF は

```
410 Gone
```

を返す。

理由

- URL は正しい  
- しかし対象リソースは削除済み  

ためである。

```
404 Not Found
```

は

- URL が不正  
- リソースの存在が不明  

な場合に使用する。

---

# 6. 業務エラー

業務チェックエラーは

```
422 Unprocessable Entity
```

を返す。

レスポンス例

```json
{
  "code": "VALIDATION_ERROR",
  "message": "入力内容に誤りがあります",
  "errors": {
    "name": "必須項目です",
    "amount": "数値を入力してください"
  }
}
```

---

# 7. クライアントのエラーハンドリング

クライアントは次のポリシーでエラーを処理する。

### 非致命エラー

```
409
410
422
```

などのエラーは

```
トースト表示
現在画面維持
```

とする。

---

### 画面遷移時エラー

Router Loader で発生したエラーは

```
トースト表示
遷移キャンセル
現在画面維持
```

とする。

---

### 致命的エラー

```
401
500
```

などは

```
グローバルエラーハンドラ
```

で処理する。

---

# 8. まとめ

BFF は HTTP ステータスコードを意味論に従って使用する。

特に更新処理では次を推奨する。

```
409 Conflict
410 Gone
422 Unprocessable Entity
```

これにより、クライアントとサーバーの責務を明確に分離できる。
