# Snapshot Model vs Reactive Model  
（SPA におけるデータ取得モデルの違い）

## 概要

SPA におけるデータ取得モデルは、大きく次の2つに分類できる。

- **Snapshot Model**
- **Reactive Model**

この2つのモデルの違いは、単にライブラリの違いではなく  
**通信対象となる API の設計モデル**の違いでもある。

| Model | 主なライブラリ | 想定する API |
|------|----------------|--------------|
| Snapshot Model | TanStack Router | Page API（BFF） |
| Reactive Model | TanStack Query | Resource API（REST） |

本ドキュメントでは、この2つのモデルの違いを整理し、  
**業務システムに適したデータ取得モデル**を説明する。

---

# API モデルの違い

SPA のデータ取得モデルは  
**通信先 API の設計**によって大きく変わる。

一般的な SPA では

```
REST Resource API
```

を前提としている。

例

```
/users
/orders
/products
```

この場合、取得単位は

```
Entity
```

となる。

```
Entity
↓
Client Cache
↓
UI
```

このモデルが **Reactive Model** である。

---

一方、本アーキテクチャでは

```
Page API
```

を採用する。

例

```
/api/orders/summary
/api/orders/detail
```

これは **BFF（Backend for Frontend）** によって提供される。

BFF とは

**画面専用の API を提供するサーバー**

である。

この場合、取得単位は

```
Page Snapshot
```

となる。

```
URL
↓
Page API
↓
Snapshot
↓
UI
```

これが **Snapshot Model** である。

---

# Snapshot Model

Snapshot Model では

```
画面 = サーバーデータのスナップショット
```

として扱う。

```
Navigation
↓
API
↓
Snapshot
↓
UI
```

画面に表示されるデータは

**ナビゲーション時点のサーバーデータ**

である。

これは従来のサーバーサイド Web アプリケーションと同じモデルである。

---

## 特徴

Snapshot Model には次の特徴がある。

- データ取得は **Navigation を起点**に行う
- 画面ごとに **1つの Snapshot** を持つ
- 画面状態は **URL で表現される**

つまり

```
URL = Application State
```

という設計になる。

---

## キャッシュ単位

Snapshot Model のキャッシュ単位は

```
Navigation Snapshot
```

である。

つまり

```
/orders
/orders/123
/orders/123?_edit=true
```

のような **URL単位**でキャッシュが管理される。

---

## 向いているシステム

Snapshot Model は次のようなシステムに適している。

- 業務システム
- CRUD中心のアプリケーション
- 検索 → 詳細 → 編集

つまり

**画面遷移中心のアプリケーション**

である。

---

## メリット

- データ整合性を保ちやすい  
- 排他制御と相性が良い  
- キャッシュ設計がシンプル  
- URL と画面状態が一致する  

---

## デメリット

- 表示後のリアクティブ更新には弱い  
- リアルタイム UI には向かない  

---

# Reactive Model

Reactive Model では

```
UI = Server State の投影
```

として扱う。

```
Server State
↓
Client Cache
↓
UI
```

画面は **サーバーデータの現在状態**を反映し続ける。

---

## 特徴

Reactive Model の最大の特徴は

**ユーザー操作を伴わないデータ更新**

を扱うことである。

つまり

```
ユーザー操作なし
↓
サーバーデータ変更
↓
UI更新
```

が発生する。

---

## 向いているシステム

Reactive Model は次のようなシステムに適している。

- SNS
- チャット
- 通知
- リアルタイムダッシュボード
- 監視システム

これらのシステムでは

**表示中の画面が他ユーザーの操作によって変化する**

ことが前提となる。

---

## キャッシュ単位

Reactive Model のキャッシュ単位は

```
Entity
```

である。

例

- User
- Order
- Product

などの **ドメインデータ単位**でキャッシュを管理する。

---

## メリット

- リアクティブ更新が容易  
- リアルタイム UI と相性が良い  

---

## デメリット

- キャッシュ設計が複雑  
- 更新画面と相性が悪い  
- 排他制御の扱いが難しい  

---

# なぜ Reactive Model を採用しないのか

TanStack Query は非常に優れたライブラリであり、  
次のようなケースでは非常に効果的である。

- リアルタイム更新が必要な UI
- 表示後もサーバーデータを継続的に同期するアプリケーション

しかし業務システムでは

- CRUD中心
- 画面遷移中心
- データ更新が頻繁

という特徴がある。

このようなシステムでは

```
画面 = サーバーデータのスナップショット
```

として扱う方が **設計がシンプルで整合性を保ちやすい。**

---

## 更新画面との相性

業務システムでは

- 閲覧モード
- 編集モード

を扱う必要がある。

特に

**楽観的排他ロック**

を使用する更新画面では

```
編集開始時のデータは必ず最新
```

である必要がある。

そのため更新画面では

```
キャッシュを使用しない
```

設計になる。

この場合 Reactive Model の利点はほとんど活かされない。

---

# 表示後データ取得の扱い

SPA では画面表示後に追加のデータ取得を行う場合がある。

例

- モーダル表示
- タブ切り替え
- ボタン操作

これらは

**URL 状態には現れない**

ため

**画面内イベント**

として扱う方が自然である。

---

## 推奨方法

このような処理では

- fetch
- ofetch
- ky

などの **シンプルな通信ライブラリ**を使用することを推奨する。

理由

- キャッシュ設計が不要
- 学習コストが低い
- 技術構成がシンプル

---

# 結論

SPA のデータ取得モデルは次の2種類に整理できる。

| Model | Fetch Unit | Cache Unit | 適したシステム |
|------|-------------|-------------|----------------|
| Snapshot Model | Page | Navigation Snapshot | 業務システム |
| Reactive Model | Entity | Entity Cache | リアルタイムUI |

---

# 本アーキテクチャの選択

本アーキテクチャでは

```
Snapshot Model
```

を採用する。

つまり

```
画面 = サーバーデータのスナップショット
```

というモデルである。

実装としては

```
TanStack Router Loader + BFF
```

を中心とした構成となる。

業務システムのように

- CRUD中心
- 更新整合性重視

のアプリケーションでは

**Snapshot Model が最も自然な設計となる。**
