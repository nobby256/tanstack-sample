# AI‑Friendly Architecture

## 概要

本アーキテクチャは **AI による設計および実装支援を前提とした構造**を採用している。

近年、Kiro をはじめとする AI 支援ツールがソフトウェア開発に広く利用されるようになっている。  
しかし現在の AI は次のような特性を持つ。

- コンテキスト長に制限がある  
- 原理からの推論は限定的である  
- 例（Example）からの類推が得意である  

そのため AI による実装支援では

**パターン化された構造**

が非常に重要になる。

本アーキテクチャは **AI が理解しやすい構造を意図的に採用している。**

---

# AI と相性の良いアーキテクチャの条件

AI 支援を前提とした設計では、次の要素が重要になる。

## パターン化された構造

AI は

```
Example → 類推
```

によってコードを生成する。

そのため **例から推測しやすい構造**が必要になる。

本アーキテクチャではすべてのページを **同一の Page Module 構造**で実装する。

```
routes/<domain>/<page>/
  route.ts
  api.ts
  page.ts
```

この構造により

- AI によるコード生成
- 実装パターンの再利用
- 開発者の理解

が容易になる。

---

# 設計と実装のトレーサビリティ

業務システムでは

**設計 → 実装**

のトレーサビリティが重要である。

本アーキテクチャでは

```
URL = Page Module
```

という関係を持つ。

```
URL
↓
routes/<domain>/<page>
↓
route.ts
↓
api.ts
↓
page.ts
```

この構造により

- 設計書
- URL
- 実装

の対応関係が明確になる。

---

# フレームワーク化された共通処理

本アーキテクチャでは次の処理を **フレームワーク層に集約する。**

- ナビゲーショントランザクション
- キャッシュ戦略
- エラーハンドリング
- API クライアント

これらはアプリケーション側で個別に実装しない。

理由は次の通り。

- AI が毎回生成する必要をなくす
- 実装のばらつきを防ぐ
- 業務システムに適した振る舞いを保証する

そのためこれらの処理は **基盤コードとして提供される。**

---

# アプリケーションコードの役割

本アーキテクチャでは、アプリケーション開発者が実装するコードを  
**業務ロジックに集中させる。**

```
route.ts → Router 設定
api.ts   → BFF API 呼び出し
page.ts  → UI 表示
```

共通処理はフレームワーク層に委譲する。

これにより

- 実装の複雑性を低減
- AI によるコード生成の安定化

が実現できる。

---

# AI と人間の役割分担

本アーキテクチャでは次の役割分担を想定する。

## AI が得意な領域

- パターン化されたコード生成
- Page Module の実装
- API 呼び出しコード

## 人間が主導する領域

- UI / UX 設計
- 業務ロジック設計
- アーキテクチャ設計
- フレームワーク基盤

特に UI / UX はプロジェクト要件の影響が強く、  
現時点では **AI より人間の判断が重要な領域**である。

そのため UI 実装は本アーキテクチャの制約対象外とする。

---

# まとめ

本アーキテクチャは

- パターン化された構造
- 設計と実装の明確な対応関係
- フレームワーク化された共通処理

を採用することで

**AI 支援と相性の良い開発モデル**

を実現する。

これにより

- AI による実装支援
- 開発者の理解
- 設計と実装のトレーサビリティ

を同時に実現できる。
