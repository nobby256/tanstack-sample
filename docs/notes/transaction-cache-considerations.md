# Transaction Cache Considerations  
（なぜトランザクションデータのキャッシュを採用しないのか）

## 概要

本ドキュメントは、一般的な SPA で採用されるキャッシュ戦略が **業務システムに必ずしも適していない理由**を説明する補足資料である。

本アーキテクチャでは **トランザクションデータのキャッシュを採用しない**。  
これは単にキャッシュを避けているのではなく、 **データ取得の順序（Render と Fetch の関係）** を重視した設計判断である。

SPA におけるデータ取得モデルは、大きく次の2種類に分類できる。

```
Fetch → Render   （Snapshot Model）
Render → Fetch   （Reactive Model）
```

本アーキテクチャでは **Fetch → Render** を採用する。これは **Navigation = Data Fetch Transaction** という設計原則を守るためである。

---

# データ取得順序の違い

データ取得モデルの違いは **Render と Fetch の順序**にある。

## Snapshot Model

```
Fetch
↓
Render
```

つまり次の順序で処理される。

```
API 呼び出し
↓
データ取得
↓
UI 描画
```

画面に表示されるデータは **その時点のサーバーデータのスナップショット**であり、従来のサーバーサイド Web アプリケーションと同じモデルである。

## Reactive Model

```
Render
↓
Fetch
↓
Re-render
```

つまり

```
UI 描画
↓
API 呼び出し
↓
UI 更新
```

という流れになる。このモデルでは **表示されたデータが最新とは限らない**。

---

# SWR（Stale‑While‑Revalidate）

TanStack Router を含む多くの SPA フレームワークで採用されている代表的なキャッシュ戦略が **SWR（Stale‑While‑Revalidate）** である。

SWR の基本動作は次の通り。

```
Cache
↓
Render
↓
Background Fetch
↓
UI 更新
```

つまり、古いキャッシュを使って画面を表示し、その後バックグラウンドで API を呼び出して **最新データで画面を書き換える**。

---

# 閲覧画面の問題

業務システムでは **信頼性の高い動作**が求められる。特に通信状態が不安定な環境では次のような問題が発生する。

- API 呼び出しの失敗  
- 通信の途中切断  
- タイムアウト  

このような場合でも **不完全な画面を表示しない**設計が重要になる。

つまり、**通信成功 = 画面遷移成功**という関係を保証する必要がある。通信が失敗した場合は空の画面や不完全な画面を表示するのではなく、**通信が不安定であることをユーザーに通知し、リトライを促す UX** を提供するべきである。

この挙動を実現するためには **Fetch → Render** という順序、すなわち **データ取得が成功してから画面を表示するモデル**が必要になる。

---

# SWR との衝突

SWR では **Render → Fetch** の順序になる。つまり

```
キャッシュ表示
↓
画面遷移完了
↓
バックグラウンド通信
```

という状態になる。

このときバックグラウンド通信が失敗した場合でも **画面は既に表示されている**。その結果、**Navigation = Transaction** というモデルを保証することが難しくなる。この理由により **閲覧画面では SWR を採用しない**という判断になる。

---

# 更新画面の問題

更新画面では多くの場合 **楽観的排他ロック**が利用される。例えば `version` フィールドなどである。更新処理では **編集開始時のデータが最新であること**が前提となる。

しかし Reactive Model では **表示されたデータが最新とは限らない**という前提になる。そのため **更新画面ではキャッシュを使用できない**設計になる。

---

# React Query との関係

React Query などのライブラリは **Entity Cache** として説明されることが多い。しかしデータ取得のタイミングを見ると実際には

```
Render
↓
Fetch
↓
Re-render
```

という **Reactive Model**であり、結果として **SWR と同じデータ取得順序**を持つ。そのため **Navigation = Data Fetch Transaction** というモデルとは相性が良くない。

---

# 最終判断

ここまでの検討を整理すると、トランザクションデータのキャッシュ戦略は次の3つに分類できる。

- Entity Cache  
- SWR Cache  
- No Cache  

しかし Entity Cache と SWR Cache はどちらも **Reactive Model**であり、閲覧画面・更新画面の双方で問題が発生する。そのため本アーキテクチャでは **トランザクションデータ → No Cache** という設計を採用する。

---

# 本アーキテクチャのキャッシュ戦略

最終的なキャッシュ戦略は次の通りである。

```
Master Data → Cache
Transaction Data → No Cache
```

つまり **キャッシュはマスタデータのためだけに使う**。

---

# まとめ

本アーキテクチャでは **Fetch → Render 型の Snapshot Model** を採用する。この設計により、データ整合性を維持しながらナビゲーショントランザクションを保証し、不完全な画面表示を防止できる。また、キャッシュ設計の複雑さを排除できる。

その結果、**Master Data Cache + Transaction Data No Cache** というシンプルで信頼性の高いキャッシュ戦略に到達する。
