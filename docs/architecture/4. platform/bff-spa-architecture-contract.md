# BFF–SPA Architecture Contract

## 概要

本ドキュメントでは **SPA クライアントと BFF（Backend for Frontend）の間で成立するアーキテクチャ上の契約（Contract）** を定義する。

この契約は

- BFF が満たすべき挙動
- SPA が前提としている挙動

を明確にするためのものである。

つまり本ドキュメントは

```
BFF の実装仕様
```

でも

```
SPA の実装仕様
```

でもなく

**SPA と BFF のインターフェース契約**

である。

---

# システム構成

本アーキテクチャでは次の構成を前提とする。

```
Browser
  ↓
BFF (Java / Servlet / Spring)
  ↓
Domain API
```

SPA は **BFF から静的リソースとして配布**される。

```
BFF
 ├ SPA static resources
 ├ Authentication
 └ Page API
```

CDN から SPA を配布する構成は想定しない。

---

# 認証の責務

認証は **すべて BFF が担当する。**

SPA は **認証済みユーザー専用アプリケーション**として動作する。

そのため SPA は次の機能を持たない。

- ログイン画面
- 認証状態管理
- トークン更新処理

これらはすべて BFF が担当する。

---

# 初回アクセスの流れ

ユーザーの初回アクセス時の動作は次の通り。

```
User
  ↓
BFF
  ↓
未認証
  ↓
ログイン画面へリダイレクト
```

ログイン成功後

```
BFF
  ↓
SPA エントリーポイントへリダイレクト
```

この設計により SPA は **常に認証済み状態で起動する。**

---

# セッション管理

認証状態は **サーバーセッション**で管理する。

```
Session Cookie
```

を使用して認証状態を保持する。

セッションが失効した場合、BFF は **HTTP 401** を返す。

---

# 認証エラーの契約

本アーキテクチャでは

```
401 = セッションタイムアウト
```

という契約を採用する。

BFF 側

```
セッションが無効
↓
HTTP 401 を返す
```

SPA 側

```
401 を受信
↓
致命的エラーと判断
↓
BFF エラー画面へ遷移
```

つまり **401 は特別なエラー**として扱われる。

---

# エラー処理の責務

本アーキテクチャでは **エラー表示の責務は SPA にある。**

ただし例外として

```
401
```

のみが特別扱いされる。

---

## 初回データ取得エラー

次のエラーは **致命的エラー**として扱う。

- 初回 Navigation の loader エラー
- beforeLoad エラー
- React render フェーズのエラー

この場合 SPA は

```
errorComponent
↓
window.location.href
↓
BFF 致命的エラー画面
```

へ遷移する。

---

## 画面遷移後のエラー

画面遷移後に発生するエラーは **すべてアプリケーションに委譲される。**

例

- API エラー
- バリデーションエラー
- ネットワークエラー

これらのエラーは **エラー通知インターフェース**に渡される。

具体的な表示方法は **アプリケーションが自由に決定できる。**

---

# 致命的エラー画面

BFF 側の致命的エラー画面では次の処理を行う。

- セッション破棄
- 自動ログアウト
- エラー画面表示

この設計の目的は **ブラウザ履歴からの再アクセス防止**である。

セッションを破棄することで

```
履歴戻り
↓
ログイン画面
```

となる。

---

# ログアウト

SPA からログアウトする場合は

**BFF のログアウトエンドポイントへ遷移する。**

ログアウト画面では次の処理を行う。

- セッション破棄
- ログアウト完了画面表示

---

# bfcache Handling

ブラウザには **Back‑Forward Cache（bfcache）** という仕組みが存在する。

bfcache は

```
ブラウザの戻る / 進む
```

操作を高速化するために、ページ全体をメモリに保存する仕組みである。

そのため次の状況が発生する可能性がある。

```
SPA
↓
ログアウト
↓
外部ページ
↓
ブラウザ Back
↓
bfcache により SPA が復元
```

この場合

```
SPA がサーバーへアクセスせずに復元される
```

可能性がある。

これは **セッション認証を採用する SPA においてセキュリティ上の問題**となる。

---

## 対策

SPA エントリーポイントでは **bfcache 復元を検知した場合にページを再読み込みする。**

例

```javascript
window.addEventListener("pageshow", (event) => {
  if (event.persisted) {
    window.location.reload()
  }
})
```

この処理により

```
bfcache 復元
↓
ページ再読み込み
↓
BFF セッション確認
↓
401 → ログイン画面
```

という安全な挙動を保証できる。

この処理は **SPA のエントリーコード（main.ts など）で一度だけ登録すればよい。**

---

# SPA の前提

本アーキテクチャでは SPA は次の前提で設計される。

**ユーザーは常に認証済みである。**

そのため SPA は

- ログイン処理
- 認証状態管理
- トークン更新

を持たない。

またフォーム入力状態は **クライアント側で管理される**。  
フォーム状態の詳細は **Form State Architecture** を参照する。

---

# まとめ

本アーキテクチャでは **認証とアプリケーションの責務を明確に分離する。**

**BFF**

- 認証
- セッション管理
- エラー画面
- SPA 配布
- Page API

**SPA**

- UI
- API 呼び出し
- エラー表示

この契約により

- セキュリティの集中管理
- 認証方式の柔軟な変更
- クライアント実装の単純化

を実現する。
