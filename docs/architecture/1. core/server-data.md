# Server Data  
（URL → Load API → Snapshot モデル）

## 概要

本ドキュメントでは **アプリケーション状態のうち Server Data（サーバーデータ）をどのように取得するか**を説明する。

Router‑First SPA Architecture ではアプリケーション状態を次の3種類に分類する。

```
Application State
├ Server Data
├ UI State
└ User Input
```

この文書では **Server Data の取得モデル**を扱う。

---

# クライアントとサーバーの役割

SPA アプリケーションは次の2つの要素で構成される。

```
Browser
↓
Client（SPA）
↓
Server（BFF）
```

クライアントの役割は比較的シンプルである。

- URL を Router に渡す  
- Loader を実行する  
- 取得したデータを描画する  

つまりクライアントは **画面データを描画するだけ**であり、  
画面データの構造は **サーバー側の設計によって決まる。**

この文書では **サーバー（BFF）側のデータ取得モデル**を説明する。

---

# 共通する基本構造

まず、従来の JSP/MVC型アプリケーションと URL Driven モデルには  
**共通している部分**がある。

サーバーロジックは

```
画面情報を取得する
↓
クライアントに渡す
↓
クライアントが描画する
```

という構造である。

```
+--------------+
|  Browser     |
+--------------+
       ↓
+--------------+
|  Client      |
|  (SPA)       |
+--------------+
       ↓
+--------------+
| Server Logic |
|   (BFF)      |
+--------------+
       ↓
+--------------+
|  Snapshot    |
+--------------+
       ↓
+--------------+
|  Render      |
+--------------+
```

---

# Load API と Event API

Server Data モデルでは API を次の2種類に分類する。

```
Load API
Event API
```

| 種類 | 役割 |
|-----|------|
| Load API | 画面を表示するためのデータ取得 |
| Event API | 操作を実行する処理 |

---

# Server Data Model

このアーキテクチャにおける Server Data の流れは次の通りである。

```
                ┌──────────────────────────┐
                │        Browser           │
                └────────────┬─────────────┘
                             │
                             │ URL
                             ▼
                ┌──────────────────────────┐
                │        Client (SPA)      │
                │  - Router                │
                │  - Loader                │
                │  - UI Rendering          │
                └────────────┬─────────────┘
                             │
            ┌────────────────┴────────────────┐
            │                                 │
            │ Load（画面取得）                 │ Event（操作）
            │                                 │
            ▼                                 ▼
   ┌────────────────────┐            ┌────────────────────┐
   │       Load API     │            │      Event API     │
   │   (画面データ取得)  │            │    (操作処理)       │
   └─────────┬──────────┘            └────────┬───────────┘
             │                                │
             ▼                                ▼
        ┌───────────────┐            ┌───────────────┐
        │  Server Logic │            │  Server Logic │
        │  (BFF)        │            │  (BFF)        │
        └───────────────┘            └───────────────┘
             │
             ▼
        ┌───────────────┐
        │   Snapshot    │
        │  (画面データ)  │
        └───────────────┘
             │
             ▼
        ┌───────────────┐
        │      UI       │
        └───────────────┘
```

---

# Load API

Load API は **画面状態を取得する API**である。

例

- 一覧画面の表示  
- 詳細画面の表示  

```
URL
↓
Loader
↓
Load API
↓
Snapshot
↓
UI
```

つまり

```
画面 = サーバーデータのスナップショット
```

として扱われる。

---

# Event API

Event API は **操作を実行する API**である。

例

- 更新
- 削除
- ダウンロード
- プレビュー

例

```
<button onclick="updateOrder()" />
```

```
updateOrder()
↓
Event API
↓
DBアクセス
```

# 例2：更新画面

更新画面は 表示 と 操作 の組み合わせである。

表示

```
クライアント

/orders/123
↓
Load API
```

操作

```
クライアント

更新ボタン
↓
Event API
```

---

# 例3：検索画面（検索条件入力 ＆ 検索結果標示）

検索画面の 検索ボタン は少し特殊である。

ユーザー視点では

```
検索ボタン
```

だが、URL Driven モデルでは

```
検索 = 自画面への画面遷移(Load)
```

として設計する。

```
クライアント

/orders
│
検索ボタン（自画面への画面遷移）
↓
/orders?keyword=abc
```

理由

```
検索結果は履歴に残るべき画面状態
```

だからである。

---

# Server Data モデルの特徴

このモデルには次の特徴がある。

- URL を起点にデータ取得する  
- API は画面単位で設計する  
- クライアントは取得したデータを表示するだけ
- 履歴に残す価値がある操作は 自画面への画面遷移(Load) として設計する  

---

# まとめ

Router‑First SPA Architecture では  
Server Data は次のモデルで取得される。

```
URL
↓
Loader
↓
Load API
↓
Snapshot
↓
UI
```

つまり

```
URL = 画面状態
Load API = 画面データ取得
Event API = 操作処理
```

という構造になる。

このモデルにより

- 画面状態の再現  
- 履歴操作  
- URL共有  

を自然に実現できる。
