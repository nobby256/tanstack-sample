# Cache Strategy

## 概要

本ドキュメントでは、本アーキテクチャにおける **キャッシュの扱い方**を定義する。

一般的な SPA では、クライアントキャッシュを中心にデータ取得戦略を設計することが多い。  
例えば次のような戦略である。

- staleTime によるキャッシュ寿命管理  
- キャッシュの invalidate  
- SWR（Stale‑While‑Revalidate）  
- エンティティキャッシュ  

しかし本アーキテクチャでは **キャッシュを主役としない設計**を採用する。

基本方針は次の通りである。

- 画面遷移では **原則として毎回 loader を実行する**
- キャッシュは **loader を実行しない遷移のための補助機構**としてのみ利用する

つまりデータ取得の基本モデルは次の通りである。

```
Navigation
↓
Loader
↓
API
↓
Snapshot
↓
UI
```

この設計を **Navigation Driven Data Fetch** と呼ぶ。

---

# 基本データ取得モデル

本アーキテクチャでは画面遷移時のデータ取得を次のように扱う。

```
Navigation
↓
Loader 実行
↓
API
↓
Snapshot
↓
UI
```

つまり

**ナビゲーションが発生したら常にサーバーからデータを取得する**

というモデルである。

この方針により

- 他ユーザー更新
- 同時更新
- 楽観的排他ロック

といった業務システム特有の整合性問題を回避できる。

---

# loader をスキップする遷移

特定の遷移では loader を実行せず、  
Router が保持しているキャッシュを利用することができる。

例

```ts
navigation.navigate({
  to: "...",
  skipLoader: true
})
```

この場合の挙動は次の通り。

```
Navigation
↓
Loader Skip
↓
Snapshot Cache 使用
↓
UI
```

つまり **loader を実行せず、前回取得したデータをそのまま表示する。**

ただし次の点に注意する。

- キャッシュが存在する場合のみ再利用される
- キャッシュが GC 済みの場合は loader が実行される

つまり

```
キャッシュはあれば使う
無ければ通常通り再取得する
```

という挙動になる。

---

# フレームワークによる Loader 制御

TanStack Router 標準のナビゲーション API では、  
**遷移単位で loader の実行／スキップを明示的に制御する機能は提供されていない。**

本アーキテクチャでは、ナビゲーション単位で loader の実行を制御するために  
**フレームワーク独自のユーティリティ**を提供する。

このユーティリティは次の機能を提供する。

- 通常のナビゲーション（loader を実行）
- loader をスキップするナビゲーション

例

```ts
const navigation = useRouteNavigation(Roote)

// 通常のナビゲーション（loader を実行）
navigation.navigate({ to: "..." })

// loaderをスキップするナビゲーション
navigation.navigate({ to: "...", skipLoader: true })
```

この仕組みにより

```
Navigation
↓
Loader 実行 / Skip
```

をアプリケーションコードから明示的に制御できる。

結果として

- 通常遷移 → 常に最新データ取得
- 特定遷移 → キャッシュ再利用

という **Navigation Driven Data Fetch** を実現する。

---

# TanStack Router におけるキャッシュ

TanStack Router は内部的に **loader の実行結果をキャッシュする**仕組みを持つ。

loader の実行は次の流れで行われる。

```
Loader
↓
API 呼び出し
↓
レスポンス取得
↓
loader の戻り値
```

このとき取得した **API のレスポンスデータ**が Router 内部に保持される。  
本ドキュメントでは、この保持されたデータを **Snapshot Cache** と呼ぶ。

重要な点は次の通り。

- キャッシュされるのは **API レスポンスの結果**
- それは **loader の戻り値として UI に渡される**
- Router はこの結果を内部キャッシュとして保持する

---

# キャッシュの寿命

TanStack Router のキャッシュは **永続的ではない**。

キャッシュの保持時間は `gcTime` によって制御される。

```
gcTime = キャッシュ保持時間
```

この時間を過ぎると、Router はキャッシュを破棄する。

そのため次の点に注意する必要がある。

- loader をスキップした遷移でも **キャッシュが存在するとは限らない**
- キャッシュが GC 済みの場合は **通常通り loader が実行される**

つまりキャッシュは

```
存在する場合だけ利用される
```

仕組みである。

---

# キャッシュの位置付け

本アーキテクチャではキャッシュを **整合性の仕組みとして使用しない。**

キャッシュは次の役割のみを持つ。

```
loader を実行しない遷移の高速化
```

そのため設計時は次の前提で考える。

- キャッシュが **存在しなくても正常に動作する**
- キャッシュは **パフォーマンス最適化としてのみ期待する**
- キャッシュの存在に依存したロジックは設計しない

つまり

```
Cache Hit → 高速表示
Cache Miss → Loader 実行
```

という程度の扱いとする。

---

# サーバーサイド Web との類似

この挙動はサーバーサイド Web における  
**戻るボタンによる画面再表示**に近い。

通常のリクエスト

```
Request
↓
Server
↓
DB
↓
HTML
```

しかし戻る操作では

```
前回描画された画面
```

が再表示される。

本アーキテクチャでは

```
loader 実行 = 新しいリクエスト
loader スキップ = 前回画面の再表示
```

という関係になる。

# 設計思想

本アーキテクチャは次の目的で設計されている。

- 更新頻度の高い業務システムを SPA で安全に実装する  
- サーバーサイド Web 技術者の設計手法を SPA に持ち込む  
- キャッシュに依存しない整合性モデルを維持する  

その上で、キャッシュは loader をスキップした遷移における  
**パフォーマンス最適化としてのみ利用する。**

本アーキテクチャでは

```
キャッシュをどう管理するか
```

ではなく

```
いつサーバーに問い合わせるか
```

を設計の中心とする。

---

# まとめ

本アーキテクチャのキャッシュ戦略は次の通り。

- 画面遷移では **原則として loader を毎回実行する**
- キャッシュは **loader をスキップした遷移でのみ利用される**
- キャッシュは **API レスポンスの結果（loader の戻り値）** を保持する
- キャッシュは `gcTime` によって自動的に破棄される
- キャッシュの存在に依存した設計は行わない

つまり

```
通常遷移 → Loader 実行
特殊遷移 → Loader Skip（キャッシュ利用）
```

というシンプルなデータ取得モデルを採用する。

この設計により

- 更新頻度の高い業務データでも整合性を維持しやすく  
- サーバー中心の設計思想を保ったまま  
- 必要な場面ではキャッシュによる高速化も利用できる

SPA アーキテクチャを実現する。
