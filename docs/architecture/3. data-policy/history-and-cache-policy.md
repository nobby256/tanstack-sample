# History and Cache Policy

## 概要

本ドキュメントでは、本アーキテクチャにおける **履歴（History）とキャッシュ（Cache）の扱い方針**を定義する。

SPA では履歴操作やキャッシュ利用がユーザー体験に大きく影響するため、多くのアプリケーションで「履歴を高速に復元するためにキャッシュを利用する」という設計が採用されることがある。

しかし業務システムでは、キャッシュによって **古いデータを再表示することが業務事故につながる可能性**がある。

そのため本アーキテクチャでは

```
履歴とは何か
キャッシュは何のために存在するのか
```

を明確に定義した上で設計方針を決定する。

---

# 履歴とは何か

ブラウザの履歴は、業務フローを保証する仕組みではない。  
履歴とは **session history 上の URL と state の移動**である。

つまり履歴操作とは

```
履歴 = URL へのジャンプ
```

である。

履歴は

- 1画面戻る
- 10画面戻る
- 履歴一覧から任意のページへ移動

といった操作を区別しない。ブラウザから見ればこれらはすべて **URL 移動**であり、アプリケーションがその意図を完全に識別することはできない。

そのため履歴を

```
過去の画面状態を完全に復元する仕組み
```

と考えるのは誤りである。

履歴の本質は

```
過去の URL へ移動するショートカット
```

である。

---

# 業務システムにおけるキャッシュのリスク

一般的な Web アプリケーションではキャッシュはパフォーマンス向上のために広く使用される。

しかし業務システムではキャッシュは **整合性リスク**を伴う。

例えば次の状況を考える。

1. 一覧画面を表示する  
2. 他ユーザーがデータを更新する  
3. 履歴操作で過去の一覧に戻る  

このときキャッシュから当時の一覧を表示すると

- 既に削除されたレコード
- 状態が変更されたレコード
- 業務条件が変わったデータ

が表示される可能性がある。

その状態で業務操作を継続すると **業務事故の原因**となる。

そのため業務システムでは

```
古いデータの復元
```

はむしろ危険な場合がある。

---

# キャッシュの基本方針

本アーキテクチャではキャッシュを **整合性の基盤として使用しない。**

キャッシュは

```
パフォーマンス最適化
```

のための仕組みであり、正しいデータを保証するものではない。

そのため基本方針は次の通りとする。

- 業務データ（トランザクションデータ）はキャッシュしない  
- 画面表示では常に最新データを取得する  
- キャッシュは主にマスターデータ共有用途に限定する  

---

# 履歴とキャッシュの関係

履歴とキャッシュはしばしば混同されるが、両者は本質的に異なる概念である。

| 概念 | 役割 |
|----|----|
履歴 | URL 移動 |
キャッシュ | パフォーマンス最適化 |

履歴は **URL による状態遷移**であり、キャッシュは **データ再利用の仕組み**である。

そのため

```
履歴 = キャッシュ
```

ではない。

履歴操作の際にキャッシュを利用する設計も存在するが、本アーキテクチャでは **履歴ではキャッシュを利用しない。**

---

# 最終方針

本アーキテクチャでは次の方針を採用する。

```
履歴操作
→ URLジャンプとして扱う
→ 常に最新データ取得
```

つまり

```
履歴 = URL 移動
キャッシュ = パフォーマンス最適化
```

として扱う。

この方針により、業務システムにおける **データ整合性を最優先した設計**を実現する。
