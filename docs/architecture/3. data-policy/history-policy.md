# History Policy

## 概要

本ドキュメントでは、本アーキテクチャにおける **ブラウザ履歴（History）の扱い方**を定義する。

SPA では履歴操作とキャッシュ利用が密接に関係することが多いが、  
本アーキテクチャでは **履歴とキャッシュを明確に分離して扱う。**

重要な原則は次の通りである。

```
履歴 = URL へのジャンプ
```

つまり履歴操作は **過去の画面状態を復元する仕組みではない。**

---

# 履歴とは何か

ブラウザ履歴は **session history に記録された URL の移動**である。

履歴操作は次のような違いを区別しない。

- 1画面戻る
- 10画面戻る
- 履歴一覧から任意のページへ移動

ブラウザから見れば、これらはすべて

```
URL へのジャンプ
```

である。

そのためアプリケーションは履歴操作の **文脈（コンテキスト）を知ることができない。**

---

# 履歴ナビゲーションの扱い

履歴操作によるナビゲーションは、通常のナビゲーションと同様に扱う。

```
History Navigation
↓
URL
↓
Router
↓
Loader
↓
API
↓
Snapshot
↓
UI
```

つまり

```
履歴操作では必ず loader を実行する
```

というポリシーを採用する。

---

# なぜ loader を必ず実行するのか

履歴操作は **コンテキストを持たない遷移**である。

例えば次の状況を考える。

```
一覧
↓
詳細
↓
更新
↓
一覧
```

ここでユーザーが履歴操作を行った場合、

```
1つ前に戻る
```

と

```
10画面前に戻る
```

は区別できない。

つまりアプリケーションは

- 更新直後の一覧なのか
- 古い履歴なのか

を判断できない。

そのため履歴ナビゲーションでは

```
必ず loader を実行し
最新データを取得する
```

という設計を採用する。

---

# loader スキップとの違い

本アーキテクチャでは **特定のナビゲーションで loader をスキップすることができる。**

例

```
更新キャンセル
↓
一覧へ戻る
```

この場合

```
navigation.navigate({ skipLoader: true })
```

として、前回の Snapshot をそのまま表示できる。

しかしこれは

```
アプリケーションが遷移の文脈を理解している場合
```

に限られる。

つまり

```
Context Navigation
```

である。

---

# Context Navigation と History Navigation

ナビゲーションは次の2種類に分類される。

## Context Navigation

アプリケーションが遷移の意味を理解しているナビゲーション。

例

- 更新キャンセル
- UI 状態変更
- 同一画面再表示

この場合は

```
Loader Skip
```

を使用できる。

---

## History Navigation

ブラウザ履歴によるナビゲーション。

例

- 戻るボタン
- 履歴一覧からの移動

この場合

```
遷移の文脈が存在しない
```

ため

```
Loader Skip は使用しない
```

---

# まとめ

本アーキテクチャでは履歴を次のように扱う。

```
履歴 = URLジャンプ
```

そのため履歴ナビゲーションでは

```
必ず loader を実行する
```

設計とする。

一方、アプリケーションが遷移の文脈を理解している場合のみ

```
Loader Skip
```

を使用する。
